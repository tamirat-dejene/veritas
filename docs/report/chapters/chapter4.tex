\section{System Design}

\subsection{Overview}
This chapter documents the system design of the Veritas online assessment platform, detailing the architectural patterns, communication models, and data tier configurations. The system adopts a hybrid communication topology, combining synchronous REST APIs for interactive actions and asynchronous Apache Kafka event queues for background workflows. Rationale for this hybrid model centers on protecting critical candidate workflows from network latency and slow downstream services (such as third-party payment APIs or machine learning loops), which is vital given Ethiopia's constrained and highly volatile network conditions. Data is isolated per service via a Database-per-Service architecture, eliminating database-level coupling and ensuring that schemas can evolve independently. Real-time proctoring and grading are assisted by AI subsystems to reduce human operational bottlenecks and accelerate scoring feedback. Within the grading subsystem, Veritas further uses a decoupled AI evaluation architecture: short-answer questions are treated as fast discriminative entailment tasks, while long-form essays are routed to a constrained generative evaluation lane.

\subsection{Specifying the Design Goals}
The system design is guided by the objectives and constraints established in Chapter 1 and the modeling artifacts in Chapter 3. The following architectural design goals were formulated, and the rationale behind each decision is as follows:

\begin{itemize}
    \item \textbf{Multi-tenant isolation:} The platform must guarantee that enterprise-specific candidate lists, assessment templates, and proctoring metrics never leak across organizational boundaries. Logical scoping by enterprise identifiers ensures data-tier security without incurring the vast infrastructure overhead of completely isolated hardware stacks.
    \item \textbf{Hybrid communication:} Synchronous REST provides immediate consistency and instant feedback for critical user operations (e.g., retrieving exam questions or submitting answers). Conversely, slow operations like dispatching email invitations or recomputing behavioral cheating scores are offloaded asynchronously to Kafka, preventing network latency or external system outages from blocking candidate-facing applications.
    \item \textbf{Scalability and fault tolerance:} During national civil service exams or university entrance assessments, the platform encounters massive surges in concurrent sessions. Centralizing rate limiting and circuit breakers at the API Gateway protects downstream microservices from CPU starvation or memory exhaustion, while asynchronous Kafka queues buffer spikes in write operations.
    \item \textbf{Integrity and auditability:} Academic and recruitment examinations are high-stakes events prone to collusion or direct database manipulation. Implementing row-level HMAC checksums and automatic PL/pgSQL database triggers enforces a strict audit trail, guaranteeing that no grade can be modified out-of-band without detection.
    \item \textbf{Security-by-design:} Operating in public, untrusted internet environments requires stateless identity validation. Implementing short-lived JSON Web Tokens (JWT) with role-based claims centralizes access controls at the gateway, avoiding the need for microservices to share session states or make repeated round-trips to a central session store.
    \item \textbf{Model-specific grading efficiency:} Automated grading should not send every subjective response to a single generative model. Short factual answers require fast concept verification, whereas essays require richer rubric-based reasoning. Separating these workloads improves latency, cost control, and evaluation consistency.
    \item \textbf{Operational portability:} Veritas must accommodate both lightweight regional test centers running on local servers and global high-availability deployments on public clouds. Standardizing on containerized microservices enables seamless transitions between Docker Compose and Kubernetes environments.
\end{itemize}

\subsection{System Design}

\subsubsection{Proposed Software Architecture}
The implemented architecture follows a highly resilient microservices pattern with an API Gateway acting as the unified ingress entry point. Rationale for selecting the microservices pattern rather than a monolithic structure includes:
\begin{itemize}
    \item \textbf{Independent Scalability:} The \texttt{candidate-service} (which handles massive candidate request concurrency during active tests) can be scaled horizontally independent of slow, heavy machine learning components like the \texttt{grading-service} or administrative tools.
    \item \textbf{Polyglot Programming:} Go was selected for performance-critical, highly concurrent edge and business services, while Python was chosen for the AI grading and face verification workloads due to its mature ML ecosystem (PyTorch, DeepFace).
    \item \textbf{Centralized Gateway Control:} The API Gateway acts as the unified boundary. Rationale for this design is to shield internal microservices from direct public access, centralize cross-cutting edge concerns (CORS, tracing, TLS termination, and DDoS protection), and prevent security token verification logic from being duplicated across internal services.
\end{itemize}
Backend services are carefully decoupled by their bounded contexts: identity authorization (\texttt{auth-service}), corporate onboarding (\texttt{enterprise-service}), test authoring and schedules (\texttt{exam-service}), active participant environments (\texttt{candidate-service}), automated AI assessment (\texttt{grading-service}), behavioral validation (\texttt{proctoring-service}), commercial licensing (\texttt{payment-service}), and asynchronous communications \linebreak (\texttt{notification-service}). The runtime model uses synchronous HTTP for immediate operations (e.g., exam content retrieval, session validation) and Kafka for asynchronous events such as enrollment invitations, exam submissions, proctoring alerts, and subscription events. This hybrid design ensures that critical user flows remain responsive while long-running or cross-service updates remain decoupled.

Key architectural principles realized in the codebase include:

\begin{itemize}
    \item \textbf{Database-per-service ownership:} Each service owns its schema and data lifecycle. Cross-service access happens only through APIs or events.
    \item \textbf{Event-driven consistency:} Kafka topics propagate lifecycle events (enterprise suspension, exam closure, grading triggers) to keep services consistent without direct coupling.
    \item \textbf{External integration boundaries:} Third-party providers (Stripe, Chapa, Cloudinary, SMTP, DeepFace, Hugging Face) are accessed only through the owning service.
\end{itemize}

\paragraph{Identity and Tenant Subsystem}
This subsystem includes \texttt{auth-service} and \texttt{enterprise-service}. Rationale for splitting identity into two services is to decouple core auth operations (which must be lightweight and operate with sub-millisecond latency using temporary tokens) from administrative onboarding and tenant metadata (which require complex, heavy PostgreSQL table relationships and audit logging). The \texttt{enterprise-service} publishes lifecycle events (suspend, reactivate, delete) to Kafka, allowing other services to lock tenant access asynchronously without locking active transactions during administrative tasks.

\paragraph{Assessment and Session Subsystem}
This subsystem includes \texttt{exam-service} and \texttt{candidate-service}. Rationale for this division is that test authoring (which is write-infrequent but requires rich templates and question-bank logic) is structurally distinct from test delivery. The \texttt{candidate-service} requires highly optimized write workflows (recording candidates' answers every few seconds to prevent loss during internet outages). It delivers questions incrementally to prevent candidates from scraping or leaking full exam packages. Upon exam submission, the candidate service emits a v3.0 slim grading trigger to Kafka rather than a massive payload, preventing message broker bloat under low-bandwidth constraints.

\paragraph{AI Proctoring and Grading Subsystem}
This subsystem contains the heavy machine learning workloads, including \texttt{proctoring-service} and \texttt{grading-service}. Rationale for isolating proctoring from grading is that the proctoring engine must ingest thousands of high-frequency behavioral signals (tab changes, blur events, webcam face checks) per candidate session, calling for high-throughput stream ingestion. Grading, on the other hand, is a high-consistency terminal execution that runs once upon exam completion. Separating these ensures that proctoring signal spikes do not resource-starve the critical scoring and grading transaction pipeline. Inside the \texttt{grading-service}, the AI workload is split again into a short-answer lane and an essay lane so that each model is matched to the structure of the assessment task.

\subsubsection{Decoupled Automated Grading Architecture}
The grading architecture avoids the anti-pattern of using one generative Large Language Model (LLM) for every subjective question. A single generative model is simple to configure, but it creates unnecessary latency for short factual responses, increases dependency on expensive external inference endpoints, and can be vulnerable to verbosity bias when wordy responses receive inflated marks. Automated scoring literature also shows that reliable grading requires both semantic understanding and explicit constraints rather than unconstrained text generation alone \citep{BenOmranAbAziz2013,Haller2022,Ramesh2022}.

Veritas therefore routes subjective responses through two task-specific lanes:
\begin{itemize}
    \item \textbf{Lane 1 -- Short-answer grading:} Short answers are treated as Natural Language Inference (NLI) and lexical alignment problems. The service compares the question context, reference answer, and candidate answer using a DeBERTa-v3-style cross-encoder, then combines the semantic score with a deterministic lemma-based Intersection-over-Union (IoU) check. This design modernizes earlier hybrid short-answer grading approaches that combined lexical overlap, sequence similarity, and semantic distance \citep{BenOmranAbAziz2013,He2023}.
    \item \textbf{Lane 2 -- Essay evaluation:} Long-form essays are routed to a local, quantized instruction-tuned Small Language Model (SLM). The essay lane decomposes the rubric into criteria such as argument quality, organization, evidence use, and clarity. Instead of accepting free-form model output, the service constrains responses to a strict JSON schema that can be parsed and audited consistently.
\end{itemize}

For short-answer grading, lexical alignment is calculated as:
\[
\text{IoU}(A,B) = \frac{|A \cap B|}{|A \cup B|}
\]
where \(A\) is the set of normalized reference lemmas and \(B\) is the set of normalized candidate-answer lemmas. The final short-answer score fuses this explicit concept overlap with the semantic cross-encoder output:
\[
\text{short\_answer\_score} =
(\alpha \times \text{semantic\_score}) +
((1-\alpha) \times \text{lexical\_iou})
\]
where \(\alpha\) is configured according to the marking policy of the question type.

For essay grading, each rubric criterion is scored independently and then mapped to a standard percentage:
\[
\text{scaled\_percentage} =
\left(
\frac{\text{average\_score} - \text{minimum\_score}}
{\text{maximum\_score} - \text{minimum\_score}}
\right) \times 100
\]
This keeps the final grade deterministic even though the individual criterion judgments are produced by a generative model. It also supports auditability because each criterion can be inspected separately instead of exposing only a single opaque grade.

This separation also improves throughput. In a typical exam with many short-answer items and only a small number of essays, the short-answer lane can batch static tensor inputs and avoid autoregressive token generation. The essay lane remains available for richer reasoning without delaying high-volume factual grading. The design therefore reduces queue pressure during mass submissions and keeps candidate result generation more predictable.

\paragraph{Commercial and Communication Subsystem}
This subsystem consists of \texttt{payment-service} and \texttt{notification-service}. Rationale for decoupling the notification service is that sending transactional emails via SMTP is slow and prone to network timeouts. Delegating notification dispatching to an asynchronous Kafka consumer prevents outbound network delays from blocking candidate logins or exam submissions. The payment service isolates Stripe and Chapa API credentials and webhook integrations, supporting parallel international and local payment providers to process subscription updates asynchronously.

\subsubsection{Database Design}
The Veritas platform implements a strict \textbf{Database-per-Service} architecture. Rationale for this design decision includes:
\begin{itemize}
    \item \textbf{Autonomy and Schema Safety:} Direct cross-service database querying or sharing tables creates coupling where a schema change in one service could break another. Forcing services to access data only through APIs ensures strict boundary contracts.
    \item \textbf{Operational Flexibility:} In the local development stack, services utilize logically isolated schemas within a single shared PostgreSQL container to conserve system memory and CPU resources. In production (Kubernetes/GKE), these schemas are physically deployed onto separate, dedicated database servers, which allows for horizontal database scaling and limits the blast radius of database failures.
    \item \textbf{Elimination of Distributed Locks:} By avoiding cross-service database joins and foreign key constraints, Veritas shifts the responsibility of data consistency to asynchronous eventual consistency models. This design prevents database locks from propagating across services, keeping individual service performance fast and deterministic.
\end{itemize}

\paragraph{Data Ownership and Consistency}
Data boundaries are explicitly defined across services to prevent distributed transaction bottlenecks and coupling:
\begin{itemize}
    \item \textbf{Auth Service Schema} (Figure \ref{fig:erd-auth}): Standardizes secure identity credentials, refresh token records, and RBAC bindings.
    \item \textbf{Enterprise Service Schema} (Figure \ref{fig:erd-enterprise}): Manages tenant metadata, staff allocations, and administrative audit logs.
    \item \textbf{Exam Service Schema} (Figure \ref{fig:erd-exam}): Stores isolated question banks, exam templates, and dynamic scheduling criteria.
    \item \textbf{Candidate Service Schema} (Figure \ref{fig:erd-candidate}): Tracks candidate enrollment tokens, active exam sessions status, and raw question-answer states.
    \item \textbf{Payment Service Schema} (Figure \ref{fig:erd-payment}): Records billing metadata, Stripe and Chapa subscription logs, and transactional invoice events.
    \item \textbf{Grading Service Schema} (Figure \ref{fig:erd-grading}): Persists final computed scores, question-by-question marks, and high-security audit rows backed by row-level cryptographic HMAC checksums.
    \item \textbf{Proctoring Service Schema} (Figure \ref{fig:erd-proctoring}): Houses facial verification logs, eye-tracking events, and aggregated cheating score histories.
\end{itemize}

Cross-service side effects are orchestrated through Kafka event consumers, ensuring eventual consistency. For instance, an enterprise suspension event triggers token revocation in Auth, templates deactivation in Exam, and session termination in Candidate services asynchronously.

\paragraph{Deployment Rationale and Notes}
\begin{itemize}
    \item \textbf{Docker Compose (Local Development Rationale):} Local development requires a lightweight, rapid feedback loop. Docker Compose starts application services and stateful infrastructure containers (Postgres, Redis, Kafka) with a single command, mounting local workspaces to support hot-reloading and avoiding the massive CPU and memory footprint of running a local Kubernetes cluster.
    \item \textbf{Kubernetes (Production Rationale):} For production environments, Kubernetes is selected to handle automated horizontal pod autoscaling (HPA) during high-stakes national exams, enable zero-downtime rolling upgrades, and provide resilient self-healing mechanisms that restart failing container instances automatically.
    \item \textbf{Kustomize \& InitContainers (Configuration Rationale):} Production manifests utilize Kustomize overlays to inject environment-specific environment configurations (dev, staging, prod) without code duplication. \texttt{initContainers} are configured on dependent service deployments to enforce sequence orchestration, ensuring that application services do not start until their target database migrations have been successfully executed by the migration runner, preventing startup panic loops.
    \item \textbf{Internal Service Discovery:} Internal communications utilize Kubernetes DNS and standard cluster service names, isolating microservices in private cluster networks to avoid exposed endpoints.
\end{itemize}

\subsection{Security Model}
This subsection documents the platform's multi-layered security architecture, detailing identity assertion, edge token validation, tenant separation, and the cryptographic and database-level audit systems.

\subsubsection{JWT Access Tokens and Claims}
Veritas enforces a token-based authentication and role-based access control (RBAC) system managed by the \texttt{auth-service}. Upon successful authentication, the service issues cryptographically signed JSON Web Tokens (JWT) using an HMAC-SHA256 signature. The token's payload contains critical authorization claims, which include:
\begin{itemize}
    \item \textbf{Subject (\texttt{sub}):} The globally unique user identifier (UUID) that identifies the authenticated user.
    \item \textbf{Enterprise ID (\texttt{enterpriseId}):} The tenant identifier (UUID) associated with the user's organization, preventing cross-tenant access.
    \item \textbf{Role (\texttt{role}):} The user's role (e.g., \texttt{super\_admin}, \texttt{admin}, \texttt{staff}, \texttt{candidate}), dictating resource access levels.
    \item \textbf{Tier (\texttt{tier}):} The active subscription tier of the enterprise (\texttt{free}, \texttt{pro}, \texttt{enterprise}), which gatekeeps commercial features.
    \item \textbf{Expiration and Issuance:} Standard registered claims (\texttt{exp}, \texttt{iat}) managing token lifecycle and prevention of replay attacks.
\end{itemize}

\subsubsection{Edge Routing and Tenant Scoping at the API Gateway}
To eliminate microservice-level token verification overhead and establish a strict security perimeter, all external HTTP requests pass through the API Gateway, which implements the following pipeline:
\begin{itemize}
    \item \textbf{Authentication Check (\texttt{JWTAuth}):} Validates the signature, expiration, and format of the incoming \texttt{Bearer} token using the shared signing secret.
    \item \textbf{Tenant Resolution (\texttt{TenantResolver}):} Extracts the \texttt{enterpriseId} from the JWT claims and injects it into the routing context.
    \item \textbf{Header Injection (\texttt{InjectUserHeaders}):} Sanitizes and converts JWT claims into standardized HTTP headers:
    \begin{itemize}
        \item \texttt{X-User-ID}: The user's UUID.
        \item \texttt{X-User-Role}: The user's authorized role.
        \item \texttt{X-Enterprise-ID}: The tenant's enterprise UUID.
    \end{itemize}
    \item \textbf{Token Stripping:} The gateway strips the original \texttt{Authorization} header before forwarding the request downstream. Downstream services do not contain JWT parsing logic; they are completely isolated inside the private virtual network and implicitly trust the \texttt{X-User-*} headers injected by the gateway, mitigating signature-spoofing and token propagation overhead.
\end{itemize}

\subsubsection{Audit Mechanisms and Row-Level Integrity}
To maintain absolute auditability and detect malicious manipulation of records, the platform incorporates dual audit strategies:
\begin{itemize}
    \item \textbf{Enterprise Service Audit Trail:} The \texttt{enterprise-service} automatically records administrative operations (e.g., onboarding, role changes, tier modifications, and suspensions) in the \texttt{veritas\_enterprise\_audit\_logs} table. This append-only log captures the actor's UUID, role, action type, specific metadata, and a precise timestamp.
    \item \textbf{Grading Service PL/pgSQL Audit Trigger:} The \texttt{grading-service} implements database-level triggers to enforce data consistency. Any \texttt{INSERT}, \texttt{UPDATE}, or \texttt{DELETE} operation on the \texttt{grading\_results} table automatically fires \linebreak the \texttt{trg\_grading\_results\_audit} trigger. The trigger extracts transaction session variables (such as actor ID, role, IP address, and change reason), compares the before and after states, and logs the change set in a separate \texttt{grading\_audit\_log} table using PostgreSQL \texttt{JSONB} for deep, queryable diffs.
    \item \textbf{Row-Level HMAC Tamper Detection:} To prevent out-of-band updates directly on database tables (e.g., bypassing application logic to alter candidate scores), each record in the \texttt{grading\_results} table features a \texttt{row\_checksum}. \linebreak The \texttt{row\_checksum} is calculated as an HMAC-SHA256 signature using a cryptographically secure \texttt{GRADING\_SECRET\_KEY} over a deterministic string concatenation of the row's immutable scoring parameters:
    \[
    \begin{aligned}
    \text{Message} ={}& \text{session\_id} \mathbin{\Vert} \text{candidate\_id} \mathbin{\Vert} \text{total\_max\_points} \\
    &\mathbin{\Vert} \text{total\_awarded\_points} \mathbin{\Vert} \text{percentage} \mathbin{\Vert} \text{version}
    \end{aligned}
    \]
    Any discrepancy between the calculated hash and the stored \texttt{row\_checksum} during read operations indicates tampering and triggers alert workflows.
    \item \textbf{Optimistic Concurrency Control:} A \texttt{version} column is updated automatically on every legitimate write. Concurrent updates targeting stale record states are automatically aborted.
\end{itemize}

\subsection{Resilience and Failure Handling}
This subsection describes the strategies implemented in Veritas to survive transient hardware, software, or network failures.

\subsubsection{API Gateway Circuit Breaker}
To prevent cascading failures when downstream microservices experience performance degradation or crashes, the API Gateway integrates \texttt{sony/gobreaker} circuit breakers. The circuit breaker operates in three states:
\begin{itemize}
    \item \textbf{Closed State:} Normal operation. All requests are routed downstream. If consecutive failures to a microservice exceed 5 (configured threshold) within a 60-second window, the circuit trips.
    \item \textbf{Open State:} The gateway immediately intercepts and rejects requests targeting the failing microservice, returning an \texttt{HTTP 503 Service Unavailable} error, thereby giving the microservice time to recover.
    \item \textbf{Half-Open State:} After a 30-second cooling-off period, the circuit enters a half-open state, allowing up to 3 test requests through. If all 3 test requests succeed, the breaker returns to the \texttt{Closed} state; if any fail, it trips back to \texttt{Open}.
\end{itemize}

\subsubsection{Resilient Kafka Consumers}
To ensure robust messaging, all event consumers (e.g., Python \texttt{aiokafka} loops in grading and proctoring services, and Go consumers in auth and notification services) are designed with a recovery pattern:
\begin{itemize}
    \item Consumers run within a persistent asynchronous loop wrapping connection logic in \texttt{try-except} (or Go's \texttt{recover}) blocks.
    \item If connection drops, authentication timeouts, or broker restarts occur, the consumer logs a structured warning and enters a 5-second backoff interval before attempting reconnection, preventing service crash-loops and self-denial of service.
\end{itemize}
